Lecture 7: Serialization and Deserialization – The Universal Language of Data

Introduction

In the previous lectures, we established how requests find their way to the server (Routing) and the methods used to express intent. However, a major hurdle remains: The Language Barrier. A JavaScript frontend and a Rust backend "think" in completely different data types. Serialization and Deserialization are the processes that bridge this gap.

1. The Core Problem: The Language Barrier

A Rust server cannot directly "read" a JavaScript memory object. To solve this, both machines must agree on a Common Standard—a format that is language-agnostic.

2. Defining the Mechanics

A. Serialization (Native → Standard)

The process of converting a native programming language structure into a standard format (e.g., JSON string) suitable for transmission or storage.

B. Deserialization (Standard → Native)

The reverse process: taking a standardized string or byte stream and parsing it back into a native data structure.

First Principle: Serialization is like translating a book into a "Universal Language" (like Morse code) so it can be sent over a wire.

3. Serialization Standards

  • Text-Based Formats: JSON (Industry standard), YAML (Configuration), XML (Older standard).
  • Binary Formats: Protobuf (Compact and fast), Avro (Big Data).

4. Deep Dive into JSON

While it originated from JavaScript, JSON is now used everywhere. Keys must be strings wrapped in double quotes.

5. End-to-End Workflow Example

  1. Client-Side: JS serializes data into a JSON string.
  2. Transmission: The string travels over HTTP as part of a POST request body.
  3. Server-Side: Backend receives request, identifies Content-Type: application/json, and deserializes body.
  4. Response: Server serializes result back to JSON and sends it to client.
